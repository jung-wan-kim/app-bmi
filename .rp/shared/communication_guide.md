# RP 간 커뮤니케이션 가이드

## 커뮤니케이션 원칙
1. **투명성**: 모든 정보는 관련 RP와 공유
2. **적시성**: 이슈 발생 시 즉시 공유
3. **명확성**: 기술 용어는 필요시 설명 첨부
4. **추적성**: 모든 결정사항 문서화

## 커뮤니케이션 채널

### 1. 정기 미팅
#### 일일 스탠드업 (매일 오전 10시)
- **참석자**: 모든 RP
- **시간**: 15분
- **형식**: 
  - 어제 한 일
  - 오늘 할 일
  - 블로커 공유

#### 주간 진행상황 리뷰 (매주 금요일 오후 3시)
- **주관**: PM_COORDINATOR
- **참석자**: 모든 RP
- **시간**: 1시간
- **안건**:
  - 주간 성과 리뷰
  - 다음 주 계획
  - 이슈 및 리스크 논의

#### 스프린트 계획 (격주 월요일)
- **주관**: PM_COORDINATOR
- **참석자**: 모든 RP
- **시간**: 2시간
- **내용**: 스프린트 목표 및 작업 할당

### 2. 기능별 미팅
#### 디자인 리뷰
- **주관**: UI_UX_DESIGNER
- **참석자**: FLUTTER_DEVELOPER, CHARACTER_DESIGNER, PM_COORDINATOR
- **주기**: 주 2회
- **내용**: 디자인 검토 및 피드백

#### 기술 리뷰
- **주관**: FLUTTER_DEVELOPER
- **참석자**: BACKEND_ENGINEER, DATA_ANALYST, DEVOPS_ENGINEER
- **주기**: 주 1회
- **내용**: 기술적 의사결정 및 아키텍처 논의

#### 품질 리뷰
- **주관**: QA_TESTER
- **참석자**: 전체 개발팀
- **주기**: 스프린트 종료 시
- **내용**: 테스트 결과 및 품질 지표 공유

### 3. 비동기 커뮤니케이션
#### Slack 채널
- **#general**: 일반 공지 및 논의
- **#dev-frontend**: Flutter 개발 관련
- **#dev-backend**: 백엔드 및 DB 관련
- **#design**: 디자인 관련
- **#qa**: 버그 리포트 및 테스트
- **#random**: 팀 빌딩 및 자유 대화

#### 문서 공유
- **Notion/Confluence**: 프로젝트 문서
- **Figma**: 디자인 파일
- **GitHub**: 코드 및 이슈 트래킹
- **Google Drive**: 기타 문서

## RP 간 주요 인터페이스

### 1. UI_UX_DESIGNER ↔ FLUTTER_DEVELOPER
- **전달 사항**: 디자인 스펙, 에셋, 스타일 가이드
- **피드백**: 구현 가능성, 성능 이슈
- **도구**: Figma, Zeplin

### 2. FLUTTER_DEVELOPER ↔ BACKEND_ENGINEER
- **전달 사항**: API 요구사항, 데이터 구조
- **피드백**: API 명세, 응답 형식
- **도구**: Postman, Swagger

### 3. DATA_ANALYST ↔ BACKEND_ENGINEER
- **전달 사항**: 데이터 분석 요구사항
- **피드백**: 데이터베이스 스키마, 쿼리 최적화
- **도구**: SQL 문서, ER 다이어그램

### 4. CHARACTER_DESIGNER ↔ FLUTTER_DEVELOPER
- **전달 사항**: 캐릭터 에셋, 애니메이션 파일
- **피드백**: 파일 크기, 성능 요구사항
- **도구**: Lottie, Rive

### 5. QA_TESTER ↔ 모든 RP
- **전달 사항**: 버그 리포트, 테스트 결과
- **피드백**: 버그 수정, 재테스트 요청
- **도구**: Jira, TestRail

## 의사결정 프로세스

### 1. 기술적 의사결정
1. 이슈 제기 (누구나 가능)
2. 관련 RP 소집
3. 옵션 검토 및 논의
4. 결정 및 문서화
5. 전체 팀 공유

### 2. 우선순위 결정
- **주관**: PM_COORDINATOR
- **입력**: 각 RP의 의견
- **기준**: 사용자 가치, 기술적 의존성, 리소스

### 3. 긴급 이슈 대응
1. 이슈 발견자가 Slack #urgent 채널에 공유
2. PM_COORDINATOR가 관련 RP 소집
3. 30분 내 대응 방안 결정
4. 실행 및 모니터링

## 문서화 규칙

### 1. 미팅 노트
- 미팅 종료 후 24시간 내 작성
- 참석자, 안건, 결정사항, 액션 아이템 포함
- Notion/Confluence에 저장

### 2. 기술 문서
- 코드와 함께 업데이트
- README, API 문서, 아키텍처 다이어그램 유지
- 변경 이력 관리

### 3. 이슈 트래킹
- 모든 작업은 티켓으로 관리
- 상태 업데이트 필수
- 완료 시 결과 문서화

## 팀 문화
- **존중**: 모든 의견을 경청하고 존중
- **협력**: 개인보다 팀의 성공 우선
- **학습**: 실패를 학습 기회로 활용
- **피드백**: 건설적이고 구체적인 피드백
- **재미**: 일을 즐기며 함께 성장
